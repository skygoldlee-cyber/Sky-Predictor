# 피봇 기반 매매 전략 (Pivot-Based Trading Strategy)

## 개요

피봇 기반 매매 전략은 KOSPI 지수의 피봇(Pivot) 확정 시점을 기반으로 KP200 선물을 매매하는 전략입니다.

- **피봇 감지**: KOSPI 지수에서 스윙 고점/저점을 감지
- **매매 종목**: KP200 선물
- **진입 시점**: 피봇 확정 시점의 KP200 선물 가격
- **청산 시점**: 반대 피봇 확정 또는 가격 돌파

---

## 전략 로직

### 진입 조건

#### 매수 (LONG) 진입
- **조건**: KOSPI 지수에서 피봇 저가(Low)가 확정되는 시점
- **진입 가격**: 그 시점의 KP200 선물 가격
- **예시**: KOSPI 지수 380.0이 저점으로 확정 → KP200 선물 382.5에 매수 진입

#### 매도 (SHORT) 진입
- **조건**: KOSPI 지수에서 피봇 고가(High)가 확정되는 시점
- **진입 가격**: 그 시점의 KP200 선물 가격
- **예시**: KOSPI 지수 385.0이 고점으로 확정 → KP200 선물 387.2에 매도 진입

### 청산 조건

#### 매수 포지션 청산
1. **목표 청산**: 다음 피봇 고가(High) 확정 시점
   - 그 시점의 KP200 선물 가격으로 청산
2. **손절 청산**: 가격이 피봇 저가 아래로 돌파 시
   - KOSPI 지수 피봇 저가를 선물 가격으로 변환하여 비교
   - KP200 선물 가격이 변환된 선물 피봇 저가보다 낮으면 손절
   - **손절 버퍼**: 슬리피지와 수수료를 고려하여 진입가 아래 일정 틱(tick)만큼 버퍼 적용 권장

#### 매도 포지션 청산
1. **목표 청산**: 다음 피봇 저가(Low) 확정 시점
   - 그 시점의 KP200 선물 가격으로 청산
2. **손절 청산**: 가격이 피봇 고가 위로 돌파 시
   - KOSPI 지수 피봇 고가를 선물 가격으로 변환하여 비교
   - KP200 선물 가격이 변환된 선물 피봇 고가보다 높으면 손절
   - **손절 버퍼**: 슬리피지와 수수료를 고려하여 진입가 위 일정 틱(tick)만큼 버퍼 적용 권장

### 연속 피봇 처리
- **동일 방향 연속 피봇**: 현재 포지션 보유 중 동일 방향 피봇(예: 매수 포지션 중 저점 확정)이 연속으로 확정될 경우
  - 추가 진입하지 않음 (현재 포지션 유지)
  - 피봇 가격 업데이트로 인한 손절 기준 재조정은 수행하지 않음
- **반대 방향 피봇**: 현재 포지션과 반대 방향 피봇 확정 시
  - 기존 포지션 청산 후 새로운 방향으로 진입 가능

### 강제 청산
- **시간**: 장 마감 시간 (기본 15:30)
- 모든 포지션 강제 청산

## KOSPI 지수와 KP200 선물의 관계

### 선물/지수 비율 (Futures/Index Ratio)
```
선물/지수 비율 = KP200 선물 가격 / KOSPI 지수 가격
```

- 이 비율은 피봇 확정 시마다 업데이트 (피봇 확정 시점의 비율 고정)
- 가격 돌파 체크 시 KOSPI 지수 피봇 가격을 선물 가격으로 변환하는 데 사용
- **참고**: 손절 체크는 피봇 확정 시점의 비율을 사용 (실시간 비율 재계산하지 않음)
  - 이유: 피봇 기준이 고정되어야 일관된 손절 기준 유지
  - 실시간 비율 변동으로 인한 손절 기준 변동 방지

### 가격 변환
```
선물 피봇 가격 = KOSPI 지수 피봇 가격 × 선물/지수 비율
```

### 예시
- KOSPI 지수 피봇 저가: 380.0
- 피봇 확정 시점 KP200 선물 가격: 382.5
- 선물/지수 비율: 382.5 / 380.0 = 1.0066
- 가격 돌파 체크 기준: 380.0 × 1.0066 = 382.5

## 백테스팅 방법론

### 백테스트 시뮬레이션 조건
- **진입/청산**: 피봇 확정 봉의 종가 사용
- **손절 기준**: 이전 피봇 확정 봉의 종가
- **지연 고려**: 피봇 확정은 최소 `confirmation_bars`만큼 지연
- **매매 기준**: KOSPI 지수 피봇
- **수익 계산**: KOSPI 지수 포인트

### 백테스트 결과 저장
- **파일명**: `pivot_backtest_YYYYMMDD.json`
- **저장 경로**: `trade_history/` (config.json에서 설정 가능)
- **저장 조건**: `strategy="pivot"`일 때만 장마감 후 자동 저장

### 실시간 시뮬레이션
- **동작**: 매 피봇 확정 시마다 실시간으로 시뮬레이션 지표 계산 및 업데이트
- **기능**:
  - 확정된 피봇 목록 관리
  - 마지막 두 피봇으로 시뮬레이션 수행
  - 실시간 승률, 총 수익 등 지표 계산
  - 텔레그램으로 실시간 시뮬레이션 결과 알림 (경고 문구 포함)
- **메서드**: `get_backtest_metrics()`로 현재 지표 조회 가능
- **신뢰성 경고**: 실시간 시뮬레이션은 샘플 크기가 작을 수 있음
  - 당일 피봇이 2~4개인 경우 승률 계산이 통계적으로 의미 없을 수 있음
  - 실시간 지표는 참고용으로만 사용하며, 장마감 후 전체 백테스트 결과가 더 신뢰 가능
  - 샤프 비율은 거래 수 3건 이상일 때만 계산 (그 외 0.0)

- **신뢰성**: 당일 거래 수가 3건 이하일 경우 샤프 비율은 참고용으로만 해석
- **공식**: `sharpe_ratio = 평균 수익 / 수익 표준편차` (표준편차 > 0일 때만)

## 설정 파라미터

### config.json 설정

```json
{
  "trade_gate": {
    "enabled": true,
    "strategy": "pivot",
    "force_close_time": "15:30",
    "market_open_time": "08:45",
    "history_dir": "trade_history",
    "sizing_method": "fixed_fractional",
    "sizing_fixed_fraction": 0.95,
    "max_daily_loss": 0.02,
    "price_check_interval_sec": 5,
    ...
  }
}
```

### 주요 파라미터

| 파라미터 | 설명 | 기본값 |
|---------|------|--------|
| `strategy` | 전략 선택 (`"signal"` 또는 `"pivot"`) | `"signal"` |
| `enabled` | 게이트 활성화 여부 | `true` |
| `force_close_time` | 강제 청산 시간 | `"15:30"` |
| `market_open_time` | 장 시작 시간 | `"08:45"` |
| `sizing_method` | 포지션 사이징 방법 | `"fixed_fractional"` |
| `sizing_fixed_fraction` | 고정 비율 사이징 (권장: 0.1~0.3, 너무 높으면 리스크 과도) | `0.95` |
| `max_daily_loss` | 일일 최대 손실 한도 (자본 대비 비율, 초과 시 신규 진입 차단) | `0.02` |
| `price_check_interval_sec` | 가격 체크 주기 (초, 권장 5초 이하) | `5` |
| `history_dir` | 이력 저장 디렉토리 | `"trade_history"` |

### 포지션 사이징 권장 범위
- **권장 범위**: `0.1 ~ 0.3` (자본의 10~30%)
- **이유**: KP200 선물의 증거금률과 레버리지 특성 고려
- **현재 설정**: `0.95`는 매우 공격적이므로 실전 운용 시 조정 권장

### max_daily_loss 동작 방식
- **한도 초과 시**: 당일 신규 진입 차단 (기존 포지션은 유지)
- **기존 포지션**: 강제 청산하지 않음 (정상 청산 조건으로 청산)
- **다음 날**: 자동 초기화 (새로운 거래일 시작 시)
- **텔레그램 알림**: 한도 초과 시 경고 알림 전송

### PivotGateConfig 파라미터

| 파라미터 | 설명 | 기본값 |
|---------|------|--------|
| `enabled` | 게이트 활성화 | `true` |
| `initial_capital` | 초기 자본 | `10000000` |
| `tick_size` | 틱 사이즈 | `0.05` |
| `commission_rate` | 수수료율 | `0.00015` |
| `slippage_ticks` | 슬리피지 틱 수 | `1` |
| `stop_loss_buffer_ticks` | 손절 버퍼 틱 수 (슬리피지 고려) | `2` |
| `max_daily_loss` | 일일 최대 손실 한도 (자본 대비 비율) | `0.02` |
| `force_close_time` | 강제 청산 시간 | `"15:30"` |

### 손절 버퍼 적용 공식
```
매수 손절 기준 = 선물 피봇 저가 - (stop_loss_buffer_ticks × tick_size)
매도 손절 기준 = 선물 피봇 고가 + (stop_loss_buffer_ticks × tick_size)
```
- **목적**: 버퍼는 손절 기준을 피봇가에서 추가로 이격시켜 노이즈성 가격 돌파를 무시
- 슬리피지와 수수료를 고려하여 일시적인 가격 변동으로 인한 조기 손절 방지
- 예: `stop_loss_buffer_ticks=2`, `tick_size=0.05`일 때 버퍼 = 0.10pt

## 구현 아키텍처

### 주요 클래스

#### PivotExecutionGate
- **위치**: `trading/pivot_gate.py`
- **역할**: 피봇 기반 진입/청산 판단
- **주요 메서드**:
  - `on_pivot_confirmed()`: 피봇 확정 시 호출
  - `check_price_breakout()`: 가격 돌파 체크 (주기적)
  - `check_close()`: 강제 청산 체크

#### PipelineTelegramBridge
- **위치**: `telegram/bridge.py`
- **역할**: 전략 선택 및 게이트 초기화
- **기능**: `strategy` 설정에 따라 TradeExecutionGate 또는 PivotExecutionGate 선택

#### MonitorsMixin
- **위치**: `telegram/monitors.py`
- **역할**: 피봇 확정 이벤트 감지 및 전달
- **주요 메서드**:
  - `_trade_monitor_loop()`: 5초 주기로 가격 체크 (급격한 가격 변동 대응)
  - `_get_latest_pivot_confirmation()`: 최신 피봇 확정 정보 추출

## 사용 예시

### 1. 전략 활성화

config.json에서 strategy 설정:
```json
{
  "trade_gate": {
    "strategy": "pivot",
    "enabled": true
  }
}
```

### 2. 코드에서 직접 사용

```python
from trading.pivot_gate import PivotExecutionGate, PivotGateConfig
from telegram.notifier import TelegramNotifier

# 설정
config = PivotGateConfig(
    enabled=True,
    initial_capital=10000000,
    force_close_time="15:30",
)

# 게이트 초기화
notifier = TelegramNotifier(...)
gate = PivotExecutionGate(notifier, config)

# 피봇 확정 시 호출
gate.on_pivot_confirmed(
    pivot_type="LOW",
    pivot_price=380.0,  # KOSPI 지수
    current_price=382.5,  # KP200 선물
)

# 주기적 가격 체크
gate.check_price_breakout(current_price=383.0)
```

## 제한 사항

1. **피봇 지연 문제 (가장 중요)**
   - 피봇 확정은 최소 `confirmation_bars`만큼 지연되며, 실제로는 그 이상 지연될 수 있음
   - 진입 시점이 이미 상당히 지난 가격일 수 있어 기대 수익 대비 리스크 비율(R:R) 왜곡 발생 가능
   - `confirmation_bars` 값에 따라 전략 성과가 크게 달라지므로 민감도 분석 및 최적화 필요
   - 지연이 클수록 진입가와 피봇가의 괴리가 커져 손절 기준이 이미 훼손된 상태에서 진입하는 경우 발생

2. **손절 기준의 모호성**
   - 현재 손절 기준은 피봇 저가/고가에 버퍼를 적용하여 노이즈성 돌파를 필터링 (구현 완료)
   - 버퍼는 손절 기준을 피봇가에서 추가로 이격시켜 일시적 가격 변동 무시
   - 슬리피지 + 수수료 고려 시 실질적 손절 기준이 진입가 위가 될 수 있음

3. **포지션 사이징 리스크**
   - `sizing_fixed_fraction: 0.95`는 자본의 95%를 단일 포지션에 사용하는 매우 공격적 설정
   - 선물 레버리지를 고려하면 실질 리스크 익스포저가 과도할 수 있음
   - 켈리 기준(Kelly Criterion) 또는 최대 드로우다운 기반 사이징 검토 권장

4. **가격 체크 주기**
   - 현재 5초 주기로 가격 체크 (구현 완료)
   - 급격한 가격 변동 시에도 빠른 대응 가능
   - 뉴스 이벤트나 서킷브레이커 상황에서도 비교적 안정적

5. **시장 리스크**: 피봇 기반 전략은 추세 시장에서 효과적이지만, 횡보 시장에서는 빈번한 손절 발생 가능
6. **슬리피지**: 실제 매매 시 슬리피지와 수수료가 수익에 영향을 미침
7. **선물/지수 비율**: 피봇 확정 시점의 비율을 고정하여 사용 (의도적 설계)
   - 실시간 비율 변동으로 인한 손절 기준 변동 방지
   - 일관된 손절 기준 유지를 위해 피봇 시점 비율 고정

## Adaptive ZigZag 메커니즘

### 개요
시스템에서 사용하는 피봇 감지 알고리즘은 Adaptive ZigZag로, 변동성에 따라 동적으로 피봇 임계값을 조정하는 메커니즘을 포함합니다.

### 핵심 기능
1. **ATR 기반 임계값 조정**
   - 평균 진폭(Average True Range)을 계산하여 시장 변동성 반영
   - 변동성이 높을 때는 피봇 임계값을 높여 노이즈 필터링 강화
   - 변동성이 낮을 때는 피봇 임계값을 낮춰 민감도 향상

2. **피봇 확정 지연 (confirmation_bars)**
   - 피봇 후보가 형성된 후 일정 봉 수(`confirmation_bars`) 동안 유지되어야 확정
   - 이를 통해 가성 피봇(False Pivot) 필터링
   - 기본값은 시장 특성에 따라 조정 필요

3. **앵커 피봇 (Anchor Pivot)**
   - 앵커 피봇은 09:00의 KOSPI 시가이며 출발점을 의미 (실제 피봇 아님)
   - 추세 기준점으로 사용되며, 이후 확정된 실제 피봇들과의 관계로 추세 판단
   - **앵커 갱신**: 앵커는 시가로 고정되며 갱신되지 않음 (추세 판단용 기준점)
   - **추세 판단**: 시가 대비 확정된 피봇들의 상대적 위치로 추세 판단
   - **ATR 연동**: ATR 기반 임계값 조정과 함께 피봇 감지 민감도 반영

### 민감도 분석
`confirmation_bars` 파라미터는 전략 성과에 큰 영향을 미치므로 다음과 같은 민감도 분석 권장:

| confirmation_bars | 특징 | 장점 | 단점 |
|------------------|------|------|------|
| 낮음 (3-5) | 빠른 피봇 확정 | 진입 기회 증가 | 가성 피봇 증가, 노이즈 민감 |
| 중간 (6-10) | 균형 잡힌 확정 | 노이즈 필터링과 기회 균형 | 일부 기회 손실 |
| 높음 (11+) | 보수적 확정 | 가성 피봇 최소화 | 진입 기회 감소, 지연 심화 |

## 장애 대응 절차

### API 연결 끊김
- **증상**: 가격 데이터 수신 불가
- **대응**:
  - 마지막 수신 가격으로 손절 체크 계속 수행
  - 일정 시간(예: 1분) 동안 데이터 없으면 포지션 강제 청산
  - 텔레그램 알림 전송

### 주문 실패
- **증상**: 진입/청산 주문 거절
- **대응**:
  - 최대 3회 재시도
  - 실패 시 텔레그램 알림 전송
  - 진입 실패 시 해당 피봇 무시
  - 청산 실패 시 신규 진입 차단 및 관리자 알림 (기존 포지션 유지)

### 데이터 지연
- **증상**: 가격 데이터 실시간성 저하
- **대응**:
  - 지연 임계값(예: 5초) 초과 시 경고 로그
  - 심각한 지연 시 포지션 보호를 위해 청산 권장

### 서킷브레이커
- **증상**: 시장 거래 정지
- **감지 방법**: 가격 데이터 수신 중단 N초(예: 30초) + 장 시간 내 → 서킷브레이커 의심
  - 가격 수신 중단과 API 연결 끊김 구분: 장 시간 내 여부로 판단
  - API 상태코드 기반 감지는 증권사 API마다 상이하므로 별도 구현 필요
- **대응**:
  - 서킷브레이커 의심 시 즉시 포지션 청산 시도
  - 청산 불가 시 텔레그램 긴급 알림 전송
